# ADR-001: Использование аппаратного NFC UID без прошивки меток

## Статус
Принято

## Контекст
По ТЗ заказчика сотрудник СК проходит маршрут и сканирует NFC-метки на контрольных точках. Отдельное архитектурное решение проекта: не перепрошивать NFC-чипы и не записывать в них служебный payload. Для идентификации достаточно считывать аппаратный UID метки.

## Решение
Backend хранит аппаратный UID в поле `nfc_tags.uid`. Это основной бизнес-ключ физической метки: по нему система находит зарегистрированную метку и связанную контрольную точку.

Поле `nfc_tags.id` остаётся внутренним UUID базы данных. Оно используется для foreign key, истории, аудита и будущей замены меток. Пользовательские сценарии не должны требовать знания этого UUID.

Поле `payload` считается необязательным и не участвует в основной логике регистрации маршрута или прохождения обхода.

## Последствия
+ Метки не нужно прошивать перед монтажом.
+ Настройка маршрута проще: настройщик сканирует метку, backend сохраняет UID и связывает его с номером точки.
+ Внутренний UUID сохраняет гибкость для истории замен и аудита.
- UID не является криптографическим секретом, поэтому антифрод должен опираться не только на UID, но и на тайминги, GPS, устройство и историю событий.
