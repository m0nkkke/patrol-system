# Модуль пользователей (Users)

Модуль users управляет пользователями системы: сотрудниками, менеджерами магазинов и администраторами сети. Пользователи не регистрируются самостоятельно: администратор создает пользователя и выдает постоянный ключ доступа.

## Эндпоинты

- `POST /api/v1/users` создаёт пользователя. Требуется JWT администратора.
- `GET /api/v1/users` возвращает список активных пользователей с пагинацией. Требуется JWT администратора.
- `GET /api/v1/users/:id` возвращает информацию об одном пользователе. Требуется JWT администратора.
- `PUT /api/v1/users/:id/shops` полностью заменяет список назначенных магазинов пользователя. Требуется JWT администратора.

## Бизнес-правила

- Поле `username` остается внутренним уникальным идентификатором и может генерироваться backend, если клиент его не передал.
- При создании пользователя backend генерирует постоянный ключ доступа формата `XXXX-XXXX-XXXX` и возвращает его в ответе.
- Ключ хранится в `users.access_key`, чтобы администратор мог видеть его в мобильной админке; для входа используется `users.access_key_hash`.
- `shopIds` содержит полный список назначенных магазинов. Все магазины проверяются до сохранения.
- `shopId` остаётся основным магазином для обратной совместимости текущего mobile-сценария.
- Если `shopId` не передан, основным становится первый элемент `shopIds`.
- Основной магазин обязательно должен присутствовать в полном списке назначений.
- Пользователь может не иметь назначенных магазинов, например администратор сети.
- API-ответы не раскрывают `password_hash` и `access_key_hash`.
- Сервисы выбрасывают доменные ошибки; преобразование в HTTP-ответы выполняется глобальным фильтром исключений.

Пример создания обходчика для двух магазинов:

```json
{
  "fullName": "Иван Петров",
  "role": "employee",
  "shopId": "00000000-0000-4000-8000-0000000000aa",
  "shopIds": [
    "00000000-0000-4000-8000-0000000000aa",
    "00000000-0000-4000-8000-0000000000bb"
  ]
}
```

## Релизная готовность mobile

- `GET /api/v1/users` поддерживает `page`, `limit`, `search`, `role`, `isActive`, `sort`. Поиск идет по ФИО и `username`; сортировки: `createdAt:desc`, `createdAt:asc`, `fullName:asc`, `fullName:desc`, `role:asc`, `role:desc`.
- `PATCH /api/v1/users/:id` обновляет ФИО, `username`, роль, активность и назначения магазинов. Если переданы `shopId` или `shopIds`, backend заново валидирует магазины и заменяет связи.
- `POST /api/v1/users/:id/access-key/rotate` перевыпускает постоянный ключ доступа пользователя и возвращает новый `accessKey` в ответе. Старый ключ после этого больше не подходит для входа.
- Все endpoints модуля доступны только роли `admin`.

Пример замены назначений:

```http
PUT /api/v1/users/:id/shops
```

```json
{
  "primaryShopId": "00000000-0000-4000-8000-0000000000bb",
  "shopIds": [
    "00000000-0000-4000-8000-0000000000aa",
    "00000000-0000-4000-8000-0000000000bb"
  ]
}
```
