# Модуль Patrols

Модуль `patrols` управляет конкретными обходами сотрудников и событиями NFC-сканирования.

## Эндпоинты

- `POST /api/v1/patrols` — стартует обход.
- `GET /api/v1/patrols/shop/:shopId` — возвращает обходы магазина с пагинацией.
- `GET /api/v1/patrols/incidents` — возвращает список инцидентов обходов с фильтрами.
- `GET /api/v1/patrols/:id` — возвращает обход с событиями.
- `POST /api/v1/patrols/:id/events` — записывает событие сканирования NFC.
- `POST /api/v1/patrols/:id/complete` — завершает обход.

## Бизнес-правила

- Обход нельзя стартовать, если в магазине нет активных контрольных точек.
- Обход нельзя стартовать, пока маршрут магазина не находится в статусе `ready`.
- При старте `total_points` фиксируется как количество активных точек магазина.
- События принимаются только для обхода в статусе `in_progress`.
- NFC UID нормализуется в нижний регистр.
- NFC-метка должна быть активной и соответствовать контрольной точке.
- Контрольная точка должна принадлежать тому же магазину, что и обход.
- После каждого события увеличивается `scanned_points`.
- Когда `scanned_points >= total_points`, обход автоматически переводится в `completed`.
- Ручное завершение разрешено только если все точки уже отсканированы.
- Первый завершённый обход магазина в корректном порядке маршрута формирует эталонные интервалы `patrol_route_intervals`.
- Для MVP допустимый коридор интервала: от 50% до 200% эталонного времени.
- Если фактический интервал меньше нижнего порога, создаётся инцидент `short_interval`.
- Если фактический интервал больше верхнего порога, создаётся инцидент `long_interval`.
- Если сотрудник сканирует точку с номером маршрута, перескочив через одну или несколько точек, создаётся инцидент `missed_point`.
- Инциденты не блокируют запись NFC-события: событие сохраняется, а подозрение фиксируется отдельно в `patrol_incidents`.

## Тайминги и инциденты

Эталон строится только один раз для магазина, когда backend впервые видит завершённый обход с событиями в порядке маршрута. Если первый завершённый обход был выполнен не по порядку, эталон не создаётся, чтобы не сохранить ошибочные интервалы.

Инциденты по таймингу привязываются к магазину, обходу, текущему событию и паре точек. Это позволяет веб-панели показывать список подозрительных обходов без пересчёта истории на лету.

## Просмотр инцидентов

`GET /api/v1/patrols/incidents` используется экраном проверяющего для просмотра подозрительных обходов.

Поддерживаемые фильтры:

- `shopId` — показать инциденты конкретного магазина.
- `employeeId` — показать инциденты конкретного сотрудника.
- `type` — `short_interval`, `long_interval`, `missed_point`.
- `from` / `to` — период по времени создания инцидента.
- `page` / `limit` — пагинация.

Ответ содержит `items`, `total`, `page`, `limit`. В `items` возвращаются связи с обходом, магазином, сотрудником, событием и точками маршрута, чтобы web-панель могла сразу показать контекст нарушения.

## Offline sync

Онлайн-запись события через `POST /api/v1/patrols/:id/events` остаётся строгой: обход должен быть `in_progress`, точка и NFC-метка должны быть активны и соответствовать друг другу.

Мобильная синхронизация через `POST /api/v1/mobile/patrols/:id/events/sync` работает мягче, потому что событие уже произошло на устройстве офлайн. Backend:

- не создаёт дубль при повторной отправке того же `localId`;
- не создаёт дубль при повторной отправке события по уже отсканированной точке обхода;
- сохраняет поздние события с флагом `late_sync = true`, не меняя `scanned_points`;
- сохраняет события по деактивированной после скана точке с флагом `point_deactivated_after_scan = true`;
- возвращает по каждому событию статус `created`, `duplicate`, `late_sync` или `point_deactivated`.
