# Модуль Patrol Schedules

Модуль `patrol-schedules` управляет повторяющимися временными окнами обходов магазина. Расписание является шаблоном: оно определяет дни недели и допустимое время старта, а конкретный обход создаётся сотрудником в мобильном приложении.

## Модель данных

Таблица `patrol_schedules` содержит:

- `shop_id` — магазин, для которого действует расписание;
- `name` — понятное название окна;
- `weekdays` — дни недели от `1` (понедельник) до `7` (воскресенье);
- `start_time` — начало допустимого окна запуска;
- `end_time` — конец окна и дедлайн завершения обхода;
- `is_active` — можно ли использовать расписание.

Время хранится без таймзоны и интерпретируется в `shops.timezone`. Ночные окна через границу суток пока не поддерживаются: `endTime` должен быть позже `startTime` в пределах одного календарного дня.

## Эндпоинты управления

Все эндпоинты требуют Bearer JWT.

- `POST /api/v1/patrol-schedules` — создать расписание; роли `admin`, `manager`.
- `GET /api/v1/patrol-schedules/shop/:shopId` — получить все расписания магазина; роли `admin`, `manager`.
- `GET /api/v1/patrol-schedules/:id` — получить расписание; роли `admin`, `manager`.
- `PATCH /api/v1/patrol-schedules/:id` — изменить расписание; роли `admin`, `manager`.
- `DELETE /api/v1/patrol-schedules/:id` — деактивировать расписание без удаления истории; роли `admin`, `manager`.

Менеджер может работать только с расписаниями своего магазина. Администратор может управлять расписаниями любого магазина.

Пример создания:

```json
{
  "shopId": "00000000-0000-4000-8000-0000000000aa",
  "name": "Вечерний обход",
  "weekdays": [1, 2, 3, 4, 5, 6, 7],
  "startTime": "20:00",
  "endTime": "21:00"
}
```

Активные окна одного магазина не могут пересекаться в общие дни недели. При пересечении backend возвращает `PATROL_SCHEDULE_OVERLAP`.

## Использование мобильным приложением

- `GET /api/v1/mobile/patrol-schedules/available` возвращает активные расписания, доступные сотруднику прямо сейчас по локальному времени его магазина.
- Каждый элемент ответа дополнительно содержит `dueAt` в UTC.
- `POST /api/v1/mobile/patrols/start` принимает необязательный `scheduleId`.
- Если `scheduleId` передан, backend повторно проверяет магазин, активность, день недели и временное окно, затем сам рассчитывает `patrols.due_at`.
- Раз в минуту backend переводит незавершённые обходы с истёкшим `due_at` в статус `overdue`.
- Статус `overdue` не блокирует маршрут: сотрудник может продолжать сканирование и завершить обход, а факт опоздания остаётся доступен для контроля.
- Если `scheduleId` не передан, создаётся внеплановый обход с `schedule_id = NULL` и `due_at = NULL`.

Расписание не создаёт `patrol` заранее. В текущей модели у расписания нет назначенного сотрудника, а `patrols.employee_id` обязателен, поэтому конкретный обход появляется в момент старта сотрудником.

## Доменные ошибки

- `PATROL_SCHEDULE_INVALID_WINDOW` — конец окна не позже начала.
- `PATROL_SCHEDULE_OVERLAP` — активное окно пересекается с другим расписанием.
- `PATROL_SCHEDULE_WRONG_SHOP` — расписание принадлежит другому магазину.
- `PATROL_SCHEDULE_INACTIVE` — расписание деактивировано.
- `PATROL_SCHEDULE_OUTSIDE_WINDOW` — попытка старта вне разрешённого дня или времени.
- `PATROL_SCHEDULE_FORBIDDEN` — пользователь не имеет доступа к магазину.
- `PATROL_SCHEDULE_DUE_AT_MANAGED` — клиент одновременно передал `scheduleId` и собственный `dueAt`.
